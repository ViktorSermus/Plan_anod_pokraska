Ответы на вопросы IT: публикация приложения для группы лиц компании

Формулировки для передачи IT и краткие пояснения для разработчика. Контекст: веб-приложение на Python (сейчас Streamlit), данные по UNC, внутренняя сеть.

1. Авторизация через Active Directory, контроллер 192.168.10.16

Суть для IT: приложение должно проверять логин/пароль пользователя в AD (или использовать уже установленную сессию Windows), используя этот DC как точку LDAP/LDAPS.

Типичные варианты

Подход Когда уместно
LDAP bind Пользователь вводит логин/пароль в веб-форме; сервер выполняет bind к AD по LDAP/LDAPS к 192.168.10.16 (или к FQDN DC).
Kerberos / SSO (IIS + Windows Auth) Пользователи в домене, браузер на ПК в той же сети — можно без ввода пароля в приложении, если стоит reverse proxy с Windows Integrated Authentication.
NTLM через IIS Аналогично, часто проще на Windows Server.

Что уточнить у IT: FQDN домена, базовый DN для поиска (DC=...), использовать ли LDAPS (порт 636) и есть ли корпоративный CA; разрешён ли доступ с сервера приложений к DC по нужным портам.

Ответ одной фразой: «Да, авторизация через AD с этим DC технически стандартна; нужны параметры LDAP/LDAPS и политика: форма логина в приложении или SSO через IIS/прокси.»

2. Авторизация через группу AD

Суть: после успешной проверки пользователя нужно разрешить доступ только членам определённой группы (например CN=Plan_anod_Users,...).

Как делается: после bind или при чтении учётной записи проверяется членство: атрибут memberOf или tokenGroups (в том числе вложенные группы), либо проверка «пользователь входит в группу X» через LDAP.

Что зафиксировать с IT: точный DN или имя группы, нужны ли вложенные группы, и что делать с пользователями не из группы (403, редирект, сообщение).

Ответ одной фразой: «Да, ограничение по группе AD — обычная практика; нужен идентификатор группы и правила для вложенных групп.»

3. Чтение файлов по UNC-путям

Суть: данные на шарах вида \\server\share\.... Процесс Python на сервере приложений должен видеть эти пути при наличии прав.

  • Служба/процесс — под учётной записью домена с правами на эти шары (часто одна service account).
  • Сеть: сервер приложений должен доходить до файловых серверов (DNS, файрвол, SMB).
  • Если по политике каждый пользователь читает только «свои» файлы под своей учёткой — сложнее: impersonation после AD-входа или общая модель ACL на шаре.

Ответ одной фразой: «Чтение по UNC поддерживается; нужно согласовать учётную запись службы (или impersonation) и сетевой доступ SMB с сервера приложений к шарам.»

4. Размещение UI на веб-сервере Flask

Важно: продукт сейчас реализован на Streamlit, а не на Flask. Flask — другой фреймворк; «перенести UI на Flask» означает по сути переписать интерфейс или строить гибрид.

Что ответить IT:

  1. Если формулировка общая («веб-интерфейс на сервере компании»): UI — веб-приложение на Python; сейчас Streamlit. Его можно опубликовать за IIS / reverse proxy на том же Windows Server — с точки зрения безопасности и URL это полноценное веб-приложение.
  2. Если IT принципиально требует именно Flask: потребуется отдельная доработка (миграция UI или фиксация в архитектуре «Python web app (Streamlit) за IIS»).
  3. Компромисс: IIS как reverse proxy + Streamlit на localhost; авторизация на IIS (Windows Auth) или в приложении (LDAP).

Ответ одной фразой: «Сейчас стек — Streamlit, не Flask; публикация как корпоративного веб-приложения на Windows Server возможна через IIS/прокси; переход на Flask — отдельный объём работ, если это обязательное требование.»

5. Операционная система Windows Server 2019

Размещение на Windows Server 2019 — нормальная схема для внутренних сервисов.

  • Python в изолированном окружении.
  • Запуск Streamlit как службы Windows (NSSM, собственная служба — по политике IT).
  • Перед приложением — IIS с ARR / URL Rewrite как reverse proxy, HTTPS, при необходимости Windows Authentication.
  • Логи, резервное копирование БД (SQLite на диске сервера или перенос на SQL Server — по решению компании).

Ответ одной фразой: «Windows Server 2019 подходит; рекомендуется IIS перед приложением, HTTPS, служба под доменной УЗ с доступом к UNC и AD.»
